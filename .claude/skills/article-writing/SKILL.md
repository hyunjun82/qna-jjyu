# 글 작성 스킬

> 키워드를 받으면 이 파일을 읽고 따라간다.
> 소상공인-확인서-발급-절차-서류-신청방법 글 수준이 최소 기준.

---

## STEP -1: 지식인 답변 먼저 작성 (사이트 글보다 먼저!)

**절대 순서: 지식인 답변 → 사이트 글. 역순 금지.**

### 답변 구조 (3단계)

**1단계: 즉답 (질문에 바로 대답)**
- 질문자가 물어본 핵심에 1~2줄로 바로 답변
- "~할 수 있어요", "~이면 돼요" (구어체)

**2단계: 핵심 정보 (소스 기반)**
- 준비물/수수료/절차 등 핵심만 요약 (리스트 형태)
- 숫자는 반드시 소스 JSON에서 가져옴
- 질문자의 세부 질문에 각각 대답

**3단계: 클릭 유도 (불완전 답변 전략)**
- 여기서 다 말하지 않는다. 사이트에 더 있다는 걸 보여줌
- "수수료 세부 내역, XX 조건, YY 서류 등 **더 자세한 내용은 여기에 정리해뒀어요**"
- → qna.jjyu.co.kr/{카테고리}/{slug} 링크

### 클릭 유도 문구 패턴
```
✅ "더 자세한 내용은 여기에 정리해뒀어요"
✅ "~별 차이, ~조건 등은 여기서 한눈에 볼 수 있어요"
❌ "제 블로그에 정리했습니다" (광고 느낌)
❌ "자세한 내용은 링크 참고" (성의 없음)
```

### 핵심 원칙
1. **즉답이 먼저** — 사이트 유도만 하면 신뢰 잃음
2. **80% 해결** — 답변만으로도 충분히 도움이 되게
3. **20% 미완** — 표, 세부 조건, 예외 등은 사이트에서만 볼 수 있게
4. **소스 명시** — "(외교부 여권안내 기준)" 등 출처 표기

## STEP 0: 소스 데이터 수집

### 절대 규칙: source-data 없이 글 작성 금지

### 수집 경로 (우선순위 순)

**경로 A: 정부/공공기관 사이트 (기본)**
```
1. Playwright → gov.kr, nts.go.kr, nhis.or.kr, nps.or.kr 등 이동
2. 본문 파싱 → JSON 변환
3. Write → source-data/{slug}.json 저장
4. Read로 저장된 JSON 확인
```

**경로 B: 공식 API (data.go.kr 등)**
```
1. API 호출 → JSON 응답
2. source-data/{slug}-api.json 저장
3. sourceUrl 필드 필수 기록
```

**경로 C: 스크린샷 캡처 (최후 수단)**
```
1. 스크린샷 → source-data/screenshots/{slug}/ 저장
2. 텍스트 정리 → source-data/{slug}-capture.json 저장
```

**경로 실패 시**: 사용자에게 "이 주제의 출처를 찾아주세요"라고 요청.

### 소스 규칙
1. source-data에 있는 내용만 글에 반영
2. 에이전트 자체 검색 정보 금지, 개인 해석/의견 금지
3. source-data에 없는 숫자/서류/기간 임의 추가 금지

## STEP 1: 글 작성

### [필수] 글 쓰기 전 — 사용자 의도 1문장

> 이 사람은 **[상황]** 상태여서 **[행동]**하려고 이 글에 들어왔다.

예: "여행가는봄 신청하려는데 어디서 하는지 몰라서 신청 경로를 찾으러 들어왔다."
이 1문장이 없으면 글 시작 금지. 이 문장이 글 전체 강조점을 결정한다.

### 메타데이터

**slug**: 한글+하이픈 (예: `코스피200-야간선물-지수-확인방법`)

**title과 h1**: 사용자가 실제로 검색창에 치는 롱테일 검색어. title=h1 동일.

```
✅ "여행가는봄 신청 어디서 하나요?"     (사용자 검색어)
✅ "여행가는봄 혜택 종류와 기간"        (사용자 검색어)
❌ "2026 여행가는봄 기간 일정 총정리"  (작성자 정리글 — "총정리" 금지)
❌ "여행가는봄 완벽 가이드"            (작성자 관점 — "완벽/총정리/정리" 금지)
```

**metaDescription**: 유일하게 문어체 허용. 160자 이내.

**description** (구어체, 140~160자): 가치 제안형.

**heroDescription** (구어체, 2~3문장):
- 첫 문장 = 독자 상황 공감 (✅ "~하려는데 어디서 하는지 모르죠?" ❌ "~알아보겠습니다")
- 마지막 문장 = 출처 신뢰도 1회 표기 ("~ {기관명} 공식 안내를 바탕으로 정리했어요.")
- **출처는 heroDescription 마지막 문장에서 1회만.** 본문 각 문단에 `(XX 기준)` 반복 금지.
- description과 90% 이상 동일 금지

**quickAnswer** (구어체, 1~2문장): 5초 안에 핵심 답변.

**날짜**: datePublished, dateModified 모두 오늘 날짜 (YYYY-MM-DD)

### 섹션 작성

섹션 3~4개. 문단 수는 내용에 따라 자유 (2~5개).

**섹션 제목** = PAA 키워드 (의문문 유지)

**핵심 원칙: 소상공인-확인서 글처럼 쓴다.**

1. **첫 문단은 즉답** — 섹션 제목의 질문에 첫 문장에서 바로 답한다.
   - ✅ "중소기업현황정보시스템(sminfo.mss.go.kr)에서 온라인으로 발급받을 수 있어요."
   - ❌ "이 제도에 대해 알아보겠습니다."

2. **혼동 방지 문단** — 헷갈리기 쉬운 것, 잘못 알기 쉬운 것을 짚는다.
   - ✅ "sbiz24.kr에도 발급 메뉴가 있지만, 그건 컨설팅 이수확인서예요. 정부 지원사업용은 반드시 sminfo에서 발급받아야 해요."
   - 이런 내용이 없으면 억지로 넣지 않는다.

3. **표 뒤에는 해석** — 표만 놓고 끝내지 않는다. "가장 먼저 확인할 건 ~예요", "특히 STEP 3에서 ~가 안 되면 ~" 등.

4. **마지막은 구체적 행동** — "1330 전화하세요"로 끝내지 않는다.
   - ✅ "sminfo.mss.go.kr → 메인 → '제출서류 안내문' 다운로드. 헷갈리면 1357로 전화해서 사업자등록번호 알려주면 안내받을 수 있어요."
   - ❌ "문의는 1330 관광콜센터로 하세요."

### 문체 규칙

- 구어체 ("~에요", "~돼요"). "~합니다/~입니다" 금지 (metaDescription 제외)
- 80대 노인도 이해하는 단어. 한 문장 80자 이내. 전문용어 괄호 설명.
- bold 뒤 콜론 금지, em dash 금지

### 사람처럼 쓰는 법 — 문장 흐름 원칙

**원칙: 각 문장이 독자의 다음 질문을 만들고, 다음 문장이 그 질문에 답한다.**

접속사("그런데", "또한", "따라서")는 이 흐름이 자연스럽게 안 이어질 때만 쓴다.
흐름이 있으면 접속사 없이도 이어진다.

```
❌ AI 패턴 (접속사로 붙이기):
"혜택은 8가지예요. 또한 신청 방법이 달라요. 따라서 표를 확인하세요."

✅ 사람 패턴 (질문-답 흐름):
"혜택이 8가지인데, 문제는 창구가 하나가 아니에요.
기차는 코레일 앱, 숙박은 야놀자, 반값여행은 visitkorea — 각각 따로예요.
내가 원하는 혜택만 아래 표에서 골라 바로 들어가세요."
```

**문장 리듬**: 짧은 → 긴 → 짧은. 짧은 문장이 포인트를 찍는다.

```
✅ "신청 창구가 없어요. [짧게 핵심]
기차는 코레일 앱에서, 숙박은 야놀자·여기어때에서, 반값여행은 visitkorea에서 —
혜택마다 경로가 달라서 한 번에 처리가 안 돼요. [길게 설명]
가장 먼저 신청해야 할 건 반값여행이에요." [짧게 행동]
```

**독자가 느껴야 하는 감정 흐름**: "아 그렇구나" → "그럼 나는 어떻게?" → "아 이렇게 하면 되겠다"

### 3대 금지
1. 정보 나열 금지 ("~이 있어요, ~도 있어요")
2. 출처 단순 번역 금지 (구어체로만 바꾸기 금지. 독자 관점으로 재구성)
3. 결론 미루기 금지 (배경부터 시작 금지)

### FAQ 작성
3~4개. **독자가 실제로 막히는 장벽**을 해소하는 질문.
- ✅ "서류가 없으면 어떻게 하나요?" (실제 장벽)
- ✅ "신용카드로도 결제 되나요?" (실제 장벽)
- ❌ "이 제도는 언제 시작되었나요?" (궁금하지 않음)
답변은 구어체 2~4문장. 출처 근거 포함.

### HTML 규칙

```html
<p>문단</p>
<table><thead><tr><th>항목</th><th>내용</th></tr></thead><tbody>...</tbody></table>
<a href="https://www.gov.kr">정부24</a>
<!-- 금지: <ul><li>, <p><strong>소제목</strong></p>, target="_blank" -->
```

**링크 앵커 텍스트 규칙 (절대)**
```
✅ <a href="https://korean.visitkorea.or.kr/travelmonth/main.do">여행가는봄 공식 사이트</a>
✅ <a href="https://www.gov.kr/...">정부24 신청 페이지</a>
❌ <a href="https://korean.visitkorea.or.kr/travelmonth/main.do">korean.visitkorea.or.kr/travelmonth/main.do</a>  (URL 그대로 금지)
❌ <a href="...">여기 클릭</a>  (의미 없는 텍스트 금지)
```
- 앵커 텍스트 = 사이트명 or 행동 ("신청 페이지", "공식 사이트", "바로가기")
- URL을 앵커 텍스트로 쓰면 verify-style.js에서 FAIL 처리됨
- 딥링크(경로 포함 URL)는 href에는 허용, 앵커 텍스트로는 절대 금지
