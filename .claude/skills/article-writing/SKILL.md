# 글 작성 스킬: 4패스 품질 파이프라인

> 키워드를 받으면 반드시 이 파일을 읽고, 순서대로 따라간다.
> 1패스(쓰고 끝) 금지. 반드시 4패스를 완주해야 커밋 가능.

---

## STEP -1: 지식인 답변 먼저 작성 (사이트 글보다 먼저!)

**절대 순서: 지식인 답변 → 사이트 글. 역순 금지.**

### 답변 구조 (3단계)

**1단계: 즉답 (질문에 바로 대답)**
- 질문자가 물어본 핵심에 1~2줄로 바로 답변
- "~할 수 있어요", "~이면 돼요" (구어체)

**2단계: 핵심 정보 (소스 기반)**
- 준비물/수수료/절차 등 핵심만 요약 (리스트 형태)
- 숫자는 반드시 소스 JSON에서 가져옴
- 질문자의 세부 질문에 각각 대답

**3단계: 클릭 유도 (불완전 답변 전략)**
- 여기서 다 말하지 않는다. 사이트에 더 있다는 걸 보여줌
- "수수료 세부 내역, XX 조건, YY 서류 등 **더 자세한 내용은 여기에 정리해뒀어요**"
- → qna.jjyu.co.kr/{카테고리}/{slug} 링크

### 클릭 유도 문구 패턴
```
✅ "더 자세한 내용은 여기에 정리해뒀어요"
✅ "~별 차이, ~조건 등은 여기서 한눈에 볼 수 있어요"
✅ "수수료 구조(기여금 포함), 미성년자 서류 등은 여기에 표로 정리했어요"
❌ "제 블로그에 정리했습니다" (광고 느낌)
❌ "자세한 내용은 링크 참고" (성의 없음)
```

### 핵심 원칙
1. **즉답이 먼저** — 사이트 유도만 하면 신뢰 잃음
2. **80% 해결** — 답변만으로도 충분히 도움이 되게
3. **20% 미완** — 표, 세부 조건, 예외 등은 사이트에서만 볼 수 있게
4. **소스 명시** — "(외교부 여권안내 기준)" 등 출처 표기

## 채점 기준표: IrrefutablePost

모든 패스에서 이 인터페이스를 기준으로 작성하고 채점한다.

```typescript
interface IrrefutablePost {
  structure: {
    ki: string;    // [기] 독자의 고통을 송곳처럼 파고듬
    seung: string; // [승] 팩트 기반 상황 전개
    jeon: string;  // [전] 통찰력 있는 비유와 반전
    gyeol: string; // [결] 100% 문제 해결 결론
  };
  readability: {
    linguistic_bridge: string; // 문장 간 논리적 연결
    universal_tone: string;    // 80대도 이해하는 쉬운 단어
  };
  intellectual_depth: {
    sharp_inquiry: string;     // 날카로운 질문
    logical_fortress: string;  // 공식 출처 근거
  };
  integrity: {
    absolute_solution: string; // 100% 문제 해결
    error_zero_check: boolean; // 오차 제로
    official_source: string;   // 공식 출처
  };
}
```

## STEP 0: 소스 데이터 수집

### 절대 규칙: source-data 없이 글 작성 금지

### 수집 경로 (우선순위 순)

**경로 A: 정부/공공기관 사이트 (기본)**
```
1. Playwright → gov.kr, nts.go.kr, nhis.or.kr, nps.or.kr 등 이동
2. 본문 파싱 → JSON 변환
3. Write → source-data/{slug}.json 저장
4. Read로 저장된 JSON 확인
```

**경로 B: 공식 API (data.go.kr 등)**
```
1. API 호출 → JSON 응답
2. source-data/{slug}-api.json 저장
3. sourceUrl 필드 필수 기록
```

**경로 C: 스크린샷 캡처 (최후 수단)**
```
1. 스크린샷 → source-data/screenshots/{slug}/ 저장
2. 텍스트 정리 → source-data/{slug}-capture.json 저장
```

**경로 실패 시**: 사용자에게 "이 주제의 출처를 찾아주세요"라고 요청.

### 소스 규칙
1. source-data에 있는 내용만 글에 반영
2. 에이전트 자체 검색 정보 금지, 개인 해석/의견 금지
3. source-data에 없는 숫자/서류/기간 임의 추가 금지

## PASS 1: Writer (초안 작성)

### 1-1. 메타데이터

**slug**: 한글+하이픈 (예: `코스피200-야간선물-지수-확인방법`)

**title과 h1**: 사용자가 제공한 PAA 키워드를 그대로 사용. title=h1 동일.

**metaDescription**: 유일하게 문어체 허용. 160자 이내.

**description** (구어체, 140~160자): 가치 제안형.

**heroDescription** (구어체, 2~3문장):
- 첫 문장 = 독자 공감 (✅ "~궁금하죠?" ❌ "~알아보겠습니다")
- 마지막 문장 = 출처 신뢰도 ("~ {기관명} 공식 안내를 바탕으로 정리했어요.")
- description과 90% 이상 동일 금지

**quickAnswer** (구어체, 1~2문장): 5초 안에 핵심 답변.

**날짜**: datePublished, dateModified 모두 오늘 날짜 (YYYY-MM-DD)

### 1-2. 섹션 작성

섹션 3~4개. 각 섹션 = 정확히 3문단 (결론 → 조건 → 실행)

**섹션 제목** = PAA 키워드 (의문문 유지)

문단 1 — 결론부터:
- ✅ "코스피200 야간선물은 Investing.com에서 실시간으로 볼 수 있어요(한국거래소 기준)."
- ❌ "이 제도에 대해 알아보겠습니다."

문단 2 — 조건·예외, 반전:
- 상황별 분기. 표 필요 시 `<table>` 사용.
- 핵심 숫자마다 공식 출처 표기 "(국세청 2026년 안내)", "(건강보험공단 기준)" 등

문단 3 — 실행 방법:
- 장소/서류/기한/링크/전화번호 구체적으로.

### 1-3. 문체 규칙

- 구어체 ("~에요", "~돼요"). "~합니다/~입니다" 금지 (metaDescription 제외)
- 문단 간 자연스러운 연결. 고정 목록에서 기계적 선택 금지
- 80대 노인도 이해하는 단어. 한 문장 80자 이내. 전문용어 괄호 설명.
- bold 뒤 콜론 금지, em dash 금지
- **연결어 다양성 규칙**:
  - 같은 글 안에서 동일 연결어 **2회까지만** (3회 이상 = FAIL)
  - 한 글에서 **최소 3개 카테고리** 연결어 사용 필수 (FAIL 조건)
  - "다만"+"반면"+"하지만"만 쓰면 FAIL. 아래에서 카테고리를 분산하세요.

| 카테고리 | 사용 맥락 | 추천 연결어 |
|----------|-----------|-------------|
| 제한·대조 | 조건/예외 | 다만, 반면, 한편, 이와 달리 |
| 이유·해석 | 근거 설명 | 실제로, 알고 보면, 그도 그럴 것이 |
| 결과·환언 | 결론 도출 | 그래서, 이에 따라, 즉, 쉽게 말해 |
| 강조·추가 | 핵심 강조 | 특히, 무엇보다, 핵심은, 심지어 |
| 예시·가정 | 사례 제시 | 예를 들어, 가령, 만약 |
| 전환·중립 | 화제 전환 | 그렇다면, 참고로, 여기서 잠깐 |
| 순서 | 절차 설명 | 먼저, 다음으로, 마지막으로 |

### 1-4. HTML 규칙

```html
<p>결론 문단</p>
<p>조건·예외 문단</p>
<p>실행 정보 문단</p>
<table><thead><tr><th>항목</th><th>내용</th></tr></thead><tbody>...</tbody></table>
<a href="https://www.gov.kr">정부24</a>
<!-- 금지: <ul><li>, <p><strong>소제목</strong></p>, target="_blank" -->
```

### 1-5. 3대 금지
1. 정보 나열 금지 ("~이 있어요, ~도 있어요")
2. 출처 단순 번역 금지 (구어체로만 바꾸기 금지. 독자 관점으로 재구성)
3. 결론 미루기 금지 (배경부터 시작 금지)

### 1-6. FAQ 작성
3~4개. 실제 사용자가 궁금해할 질문. 답변은 구어체 2~4문장. 출처 근거 포함.

### 1-7. 시각화 매핑

## PASS 2: Critic (독립 채점) — subagent 필수

subagent에게 8규칙 채점 요청:

| 규칙 | 기준 |
|------|------|
| 1. 스토리텔링 | 3~4섹션, 3문단(결론→조건→실행), 나열 금지 |
| 2. 기승전결 | 기(공감)→승(데이터)→전(반전)→결(행동) |
| 3. 문체 | 구어체, AI냄새 0건, 어미 다양성 |
| 4. 날카로운 논리 | 의문문 제목, 공식 출처 근거, 연결어 다양성 |
| 5. 가독성 | 문장 20~65자, 80자 초과 3건 이하, 괄호 설명 |
| 6. 문제해결 | 기관/기한/서류/링크/연락처 완비 |
| 7. 오차 제로 | HTML 균형, 모호 표현 금지, 내부 모순 없음 |
| 8. 출처 | source-data 존재, hero 출처 문구, 숫자 일치 |

판정: 7점 미만 항목 → FAIL → Pass 3 | 평균 8점+ → PASS → Pass 4

## PASS 3: Refiner (수정)

7점 미만 항목만 수정. 잘 된 부분은 건드리지 않음.

## PASS 4: Judge (최종 판정) — subagent 필수

8규칙 전항목 7점+, 평균 8점+, 팩트 불일치 0건 → PASS

